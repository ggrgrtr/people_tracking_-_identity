1. Назначение программы

Файл `2.py` является точкой входа в новый проект по отслеживанию и идентификации людей.
Он запускает пакет `identity_gallery_tracking`, в котором реализована более правильная
архитектура, чем в старом проекте: временное отслеживание движения человека отделено
от долгоживущей идентичности человека.

Главная цель программы:
- обнаруживать людей на кадре с помощью YOLO;
- отслеживать их положение между кадрами;
- пытаться присваивать одному и тому же человеку постоянный ID;
- сохранять историю его маршрута;
- поддерживать более плавную работу веб-камеры за счет оптимизаций.


2. Как устроен проект

Точка входа:
- `2.py` -> запускает `identity_gallery_tracking.app.main()`

Основные файлы проекта:
- `identity_gallery_tracking/app.py`
  Главный цикл программы. Открывает камеру или видео, читает кадры, запускает детектор,
  трекер, identity manager, отрисовку и сохранение логов.

- `identity_gallery_tracking/config.py`
  Все основные настройки: размеры камеры, интервалы YOLO, размеры ReID-входа, пороги
  идентификации, путь к выходным данным и т.д.

- `identity_gallery_tracking/tracklets.py`
  Здесь хранится логика краткоживущих треков, то есть временных объектов отслеживания
  на последовательности кадров. Tracklet отвечает за рамку, сглаживание, путь движения,
  Kalman-прогноз и сопоставление между соседними кадрами.

- `identity_gallery_tracking/identity_manager.py`
  Долгоживущая память системы. Этот модуль хранит уже не временные треки, а именно
  "личности" людей, их галереи признаков и историю маршрутов. Именно здесь принимается
  решение, является ли текущий tracklet тем же человеком, который уже был раньше.

- `identity_gallery_tracking/face_backend.py`
  Опциональный модуль для идентификации по лицу. Если в проект будут добавлены модели
  YuNet и SFace в формате ONNX, система сможет использовать лицо как более сильный признак.
  Если моделей нет, модуль остается отключенным и не влияет на работу.

- `identity_gallery_tracking/renderer.py`
  Отвечает за отрисовку рамок, ID, траектории, информационной панели на экране.

Повторно используемые модули из старого проекта:
- `people_tracking/detector.py`
  YOLO-детектор людей.
- `people_tracking/reid.py`
  Вычисление ReID-признаков и цветовых дескрипторов.
- `people_tracking/utils.py`
  Общие утилиты: построение путей сохранения, FPS-метр, bbox-операции и т.д.
- `people_tracking/assignment.py`
  Алгоритм Hungarian для сопоставления.


3. Ключевая архитектурная идея

Старый подход:
- "один трек = один человек"

Новый подход:
- "tracklet" = временный объект, который живет, пока человека удается более-менее
  непрерывно вести между кадрами;
- "identity" = долгоживущая сущность, которая хранит память о человеке даже после
  его выхода из кадра.

Это важно, потому что:
- трек может оборваться из-за перекрытия, выхода из кадра, плохой детекции;
- но личность человека при этом может быть сохранена в галерее признаков;
- при повторном появлении нового tracklet система пытается не выдать новый ID сразу,
  а сопоставить его с уже существующей личностью.


4. Полный поток обработки кадра

Ниже приведена последовательность работы программы.

4.1. Запуск

При запуске программа:
- считывает аргументы командной строки;
- загружает конфигурацию `AppConfig`;
- открывает источник видео: веб-камеру или видеофайл;
- определяет устройство выполнения: `cuda` или `cpu`;
- создает:
  - YOLO-детектор;
  - ReID-энкодер;
  - optional face backend;
  - tracklet tracker;
  - identity manager.

4.2. Захват кадров

Если источник является веб-камерой, включается потоковый захват:
- отдельный поток читает кадры из камеры;
- основной поток занимается обработкой;
- это уменьшает подтормаживания, потому что тяжелая обработка не блокирует чтение кадра.

Если источник - видеофайл:
- кадры читаются из основного цикла без отдельного потока.

4.3. Оценка FPS

Программа измеряет фактическую скорость поступления кадров:
- для камеры FPS считается по реальному интервалу между чтениями кадров;
- для файла используется метаинформация или временные метки.

На экран выводится:
- `FPS`
- количество активных tracklet-ов
- количество tracklet-ов, которым уже присвоен постоянный identity ID
- индикатор `REC`, если включена запись

4.4. Решение: делать ли новую детекцию

YOLO не запускается на каждом кадре.
Программа использует интервалы:
- `yolo_interval` - если активные tracklet-ы уже есть;
- `empty_scene_yolo_interval` - если сцена пустая.

Это снижает нагрузку и дает более плавную картинку.

Если кадр не является кадром детекции:
- выполняется только прогноз tracklet-ов через Kalman;
- рамки продолжают двигаться по предсказанию;
- экран обновляется быстрее.

4.5. Детекция людей

Если текущий кадр выбран как детекционный:
- кадр уменьшается;
- запускается YOLO;
- выбираются только объекты класса "person";
- слишком маленькие рамки отбрасываются;
- дублирующиеся рамки подавляются.

4.6. Адаптивный ReID

ReID - одна из самых тяжелых частей программы.
Поэтому он работает не на каждом detect-кадре одинаково.

Система смотрит:
- есть ли уже активные tracklet-ы;
- есть ли tracklet-ы без присвоенной личности;
- много ли новых детекций;
- какие детекции реально нуждаются в уточнении признаков.

После этого вычисление признаков выполняется:
- либо для всех нужных людей;
- либо только для части детекций;
- либо пропускается, если кадр не требует дорогостоящей идентификации.

В ReID используется:
- embedding внешнего вида;
- цветовой дескриптор одежды;
- при наличии face backend - признак лица.

4.7. Tracklet tracking

Модуль `tracklets.py` выполняет:
- прогноз положения через Kalman Filter;
- сопоставление текущих детекций с существующими tracklet-ами;
- Hungarian assignment для глобального сопоставления;
- обновление bbox, пути и признаков;
- создание новых tracklet-ов для новых детекций;
- удаление умерших tracklet-ов;
- удаление дубликатов tracklet-ов.

Tracklet хранит:
- текущую и сглаженную рамку;
- путь движения;
- embedding внешнего вида;
- цветовой дескриптор;
- face embedding, если доступен;
- shape descriptor;
- статистику попаданий и пропусков.

4.8. Identity assignment

После обновления tracklet-ов модуль `identity_manager.py` пытается понять,
какому человеку принадлежит tracklet.

Идея:
- если tracklet еще не готов, он остается без identity ID;
- если tracklet уже стабилен, система пробует сопоставить его с существующими личностями;
- если хорошего совпадения нет, создается новая личность с новым ID.

При сравнении используются:
- face similarity, если доступны признаки лица;
- ReID similarity;
- color similarity;
- shape similarity;
- штраф за слишком большой временной разрыв.

Если у кандидатов слишком близкие оценки:
- система предпочитает не делать уверенный матч;
- это снижает риск случайно выдать одному человеку чужой ID.

4.9. Обновление identity-галереи

Identity manager хранит галереи признаков для каждого человека:
- ReID-векторы;
- цветовые признаки;
- face-векторы;
- shape-признаки;
- историю маршрута.

Важно:
- галерея теперь обновляется не на каждом predict-only кадре;
- она обновляется только тогда, когда действительно пришла новая детекция.

Это сделано по двум причинам:
- меньше лишних вычислений;
- меньше засорение галереи повторяющимися и шумными признаками.

4.10. Отрисовка

На финальном кадре рисуются:
- рамка вокруг человека;
- либо `Track N`, если постоянный ID еще не выдан;
- либо `ID N`, если человек уже сопоставлен с identity;
- линия траектории движения;
- стрелка направления;
- панель с FPS и служебной информацией.

Если людей нет:
- выводится красная надпись `No people detected`.

4.11. Завершение

После остановки программы:
- все оставшиеся активные tracklet-ы финализируются;
- identity manager сохраняет лог;
- для каждого identity строится PNG-изображение маршрута;
- при необходимости сохраняется выходное видео.


5. Какие данные используются для идентификации

Программа не использует один признак.
Она строит решение на комбинации нескольких источников:

5.1. ReID embedding

Это основной признак внешнего вида человека:
- формируется нейросетью на основе crop-а человека;
- подходит для сравнения одежды, силуэта, общего внешнего вида;
- хорошо работает, когда человек виден целиком и достаточно крупно.

5.2. Color descriptor

Дополнительный признак:
- строится по HSV-гистограммам;
- учитывает общий цвет одежды;
- дешевле, чем глубокий embedding;
- но сам по себе недостаточно надежен.

5.3. Shape descriptor

Вспомогательный признак:
- относительная ширина, высота и соотношение сторон bbox;
- помогает отсечь явно неподходящих кандидатов;
- не предназначен для самостоятельной идентификации.

5.4. Face descriptor

Самый сильный признак в архитектуре, но только при наличии моделей лица.
Сейчас он опционален.

Если в проект добавить:
- `face_detection_yunet_2023mar.onnx`
- `face_recognition_sface_2021dec.onnx`

то система сможет:
- обнаруживать лицо внутри рамки человека;
- строить face embedding;
- сильнее различать похожих по одежде людей.


6. Почему новая архитектура лучше старой

Преимущества нового проекта:
- временное отслеживание отделено от долговременной личности;
- личность хранит свою галерею признаков;
- можно развивать face backend отдельно;
- можно развивать tracklet tracking отдельно;
- проще добавлять новые признаки и новые правила сопоставления;
- проще отлаживать ошибки "трек потерялся" и "личность перепуталась".

То есть:
- старый проект отвечал на вопрос "где сейчас объект";
- новый проект пытается отвечать на два вопроса:
  1. где сейчас объект;
  2. тот же ли это человек, что был раньше.

8. Основные оптимизации, уже встроенные в программу

В проект уже встроены следующие оптимизации:

- Потоковый захват камеры.
  Кадры читаются отдельно от тяжелой обработки.

- Детекция не на каждом кадре.
  На части кадров работает только Kalman prediction.

- Адаптивный ReID.
  Глубокие признаки считаются только там, где они действительно нужны.

- Частичный face backend.
  Если face backend отключен, он не тратит время на пустые проходы.

- Ограничение числа ReID-кандидатов.
  Не каждая детекция отправляется в тяжелую нейросеть.

- Обновление identity-галереи только на детекционных кадрах.
  Это уменьшает нагрузку и шум.

- Подавление дублей.
  Дублирующиеся рамки и tracklet-ы объединяются или отбрасываются.


9. Какие параметры сильнее всего влияют на скорость

Файл:
- `identity_gallery_tracking/config.py`

Самые важные параметры:

- `camera_width`, `camera_height`
  Чем меньше разрешение, тем быстрее работает программа, но тем меньше деталей
  для идентификации.

- `yolo_interval`
  Чем больше значение, тем реже запускается YOLO и тем выше скорость.
  Но слишком большое значение ухудшает реакцию на новые объекты.

- `empty_scene_yolo_interval`
  Влияет на скорость в пустой сцене.

- `yolo_scale`
  Дополнительное уменьшение кадра перед детекцией.

- `yolo_imgsz`
  Размер входа YOLO. Меньше = быстрее, но хуже точность детекции.

- `reid_input_height`, `reid_input_width`
  Размер входа в ReID-модель.

- `reid_interval`
  Чем больше, тем реже считается embedding.

- `max_reid_detections`
  Сколько детекций за один detect-проход допускается отправить в тяжелый ReID.

- `threaded_camera_capture`
  Включает отдельный поток чтения камеры.


10. Ограничения текущей версии

Несмотря на улучшенную архитектуру, важно понимать ограничения:

- Если в проекте нет face-моделей, идентификация идет в основном по внешности,
  одежде, цвету и размеру человека.

- Если два человека одеты похоже, находятся далеко от камеры или плохо освещены,
  их все равно можно перепутать.

- При низком разрешении веб-камеры человек в кадре может содержать слишком мало
  деталей для качественного различения.

- ReID по внешности не является биометрическим распознаванием личности.

То есть:
- новая архитектура лучше для поддержки устойчивых ID;
- но она не гарантирует абсолютного распознавания "кто именно перед камерой".

13. Итог

Данный проект представляет собой модульную систему
"детекция + tracklet tracking + identity gallery".

Его сильные стороны:
- более правильная архитектура;
- лучшее разделение ответственности между модулями;
- возможность дальнейшего развития;
- поддержка постоянных ID лучше, чем в старом варианте;
- сохранение истории маршрутов и логов.

Если развивать проект дальше, наиболее полезные направления:
- подключение face-моделей;
- отдельный режим "fast webcam" и "accurate identity";
- отладочный режим с выводом similarity score;
- тонкая настройка порогов под конкретную камеру;
- дообучение ReID-модели под вашу сцену.
