2. Главная идея проекта

Ключевая архитектурная мысль программы:

- `tracklet` не равен `identity`;
- `tracklet` - это временная сущность, живущая пока объект удается непрерывно сопровождать по соседним кадрам;
- `identity` - это долговременная сущность, которая хранит память о человеке после разрыва трека.

Именно это отделение позволяет решить две разные задачи:

- локальная задача:
  "где находится человек сейчас и как движется его прямоугольник в ближайших кадрах";

- долговременная задача:
  "тот ли это человек, которого система уже видела раньше".

Если использовать только обычный multi-object tracking, то после потери трека человек часто получает новый ID.
В данном проекте поверх треков добавлена identity gallery, то есть галерея признаков, которая пытается
переиспользовать старый ID для нового tracklet.


3. Общий конвейер обработки

Программа работает как последовательность этапов:

1. Открывает источник.
   Это либо веб-камера, либо видеофайл.

2. Оценивает режим работы.
   Для камеры может использоваться потоковый захват кадров, для файла обычно достаточно синхронного чтения.

3. Решает, нужен ли очередной детекционный проход.
   YOLO запускается не на каждом кадре, а через интервалы, чтобы снизить нагрузку.

4. Если детекция нужна:
   - ищет людей YOLO-моделью;
   - удаляет дубликаты детекций;
   - адаптивно выбирает, для каких детекций считать дорогие ReID-признаки;
   - при наличии face backend пытается извлечь face embedding.

5. Обновляет tracklet tracker.
   - существующие треки сначала прогнозируются Kalman Filter;
   - затем выполняется сопоставление detection <-> tracklet через Hungarian assignment;
   - совпавшие треки обновляются измерением;
   - пропущенные треки помечаются как missed;
   - новые детекции превращаются в новые tracklet.

6. Обновляет identity manager.
   Для стабилизировавшихся tracklet система пытается:
   - сопоставить их с уже существующими identity;
   - либо создать новую identity, если уверенного совпадения нет.

7. Рисует результат.
   На кадр накладываются рамки, ID, траектория, счетчики и текущий FPS.

8. Сохраняет артефакты сессии.
   Итоговая сессия может содержать:
   - JSON-лог identities;
   - видео с отрисовкой;
   - PNG-маршруты по каждой identity.


4. Детекция людей

Детекция реализована в `people_tracking/detector.py`.

Используемый метод:

- YOLO из пакета `ultralytics`;
- класс `person` only;
- детекция идет по уменьшенной версии кадра для экономии времени.

Применяемые механизмы:

- `yolo_scale`
  кадр уменьшается перед подачей в детектор, после чего координаты масштабируются обратно;

- `yolo_imgsz`
  отдельный параметр размера входа YOLO;

- `yolo_conf`
  минимальный порог уверенности детекции;

- `min_box_w`, `min_box_h`
  слишком маленькие рамки отсеиваются сразу как шум;

- suppression of duplicates
  после детекции рамки проходят собственное подавление дублей.

Подавление дублей детекций строится на двух критериях:

- `IoU`
  если пересечение достаточно велико, одна из рамок отбрасывается;

- `nested overlap`
  если одна рамка почти вложена в другую и центр лежит внутри более крупной рамки, это тоже считается дублем.

Почему это важно:

- YOLO иногда возвращает несколько близких person-box для одного человека;
- если такие рамки оставить, трекер начнет плодить дублирующиеся tracklet;
- дальше это ухудшит и tracking, и re-identification.


5. Почему детекция не запускается на каждом кадре

Полный detect-pass - это одна из самых тяжелых операций в конвейере.
Поэтому проект использует "разреженную" детекцию:

- `yolo_interval`
  период детекции, когда в сцене уже есть активные tracklet;

- `empty_scene_yolo_interval`
  более редкий или более частый интервал для пустой сцены без активных tracklet.

На кадрах без детекции происходит только:

- Kalman prediction;
- сглаживание bbox;
- отрисовка прогнозируемого положения.

После последних исправлений identity-логика обновляется только по реально наблюденным кадрам,
а не по голым Kalman-прогнозам. Это устраняет накрутку `observations`, `distance_px` и route image
на кадрах, где человек фактически уже не детектировался.


6. Appearance ReID: внешний вид человека

Модуль `people_tracking/reid.py` отвечает за извлечение признаков внешности.

Используемая стратегия:

- backbone:
  `ResNet50`;

- global pooling:
  `GeM pooling`;

- part-based descriptor:
  горизонтальное разбиение карты признаков на несколько частей;

- final descriptor:
  конкатенация глобального признака и part-based признаков с последующей L2-нормализацией.

Это дает более информативный embedding, чем простой average pooling:

- глобальная часть улавливает общий внешний вид;
- part-based часть помогает лучше различать верх/низ одежды и локальные детали.

Настраиваемые параметры:

- `reid_num_parts`
  сколько горизонтальных частей использовать;

- `reid_global_weight`
  вес глобального признака;

- `reid_part_weight`
  вес part-based части;

- `reid_gem_p`
  параметр GeM pooling.

Логика загрузки весов:

- если найден `reid_resnet50_msmt17.pth` или другой совместимый checkpoint, используется он;
- если специализированные ReID-веса не загрузились, модель откатывается к ImageNet-only варианту.

Это важно для эксплуатационной устойчивости:

- проект не обязан падать, если нужный checkpoint отсутствует;
- но без специализированных ReID-весов качество постоянных ID будет хуже.


7. Как именно строится ReID embedding

Шаги для одной детекции:

1. Из кадра вырезается crop человека.
2. К crop может добавляться padding (`reid_padding`), чтобы не терять края одежды.
3. Crop приводится к размеру `reid_input_width x reid_input_height`.
4. Выполняется нормализация по mean/std в стиле ImageNet.
5. Пакет crop'ов прогоняется через backbone батчем.
6. На выходе получаем L2-нормированный descriptor.

Преимущества такой схемы:

- устойчивость к небольшому изменению масштаба;
- возможность батчевой обработки нескольких детекций;
- совместимость с классическим cosine similarity.


8. Цветовой дескриптор одежды

Кроме глубокого embedding проект считает более дешевый цветовой признак.

Он строится так:

- crop переводится в HSV;
- считаются гистограммы по Hue и Saturation;
- отдельно анализируются:
  - весь человек;
  - верхняя часть;
  - нижняя часть;
- в признак дополнительно добавляются средние значения насыщенности и яркости.

Итоговый color descriptor помогает в случаях, когда:

- у двух кандидатов похожая геометрия;
- глубокий embedding не очень уверенный;
- одежда имеет достаточно характерную цветовую структуру.

Параметры:

- `reid_hist_h_bins`
- `reid_hist_s_bins`

Сравнение идет не по одному критерию, а по смеси:

- histogram intersection;
- L1-based similarity.

Это дает более стабильное сравнение, чем один только cosine similarity над цветовым вектором.


9. Shape descriptor

Дополнительный слабый, но полезный признак - shape descriptor.

Он строится из bbox относительно размеров кадра:

- относительная ширина;
- относительная высота;
- aspect ratio.

Shape descriptor не предназначен для самостоятельной идентификации человека.
Его роль:

- отсекать явно неподходящие совпадения;
- немного усиливать устойчивость матчинга при близких appearance-score;
- помогать в условиях слабого ReID-сигнала.


10. Tracklet tracking

Tracklet tracking реализован в `identity_gallery_tracking/tracklets.py`.

Каждый `Tracklet` хранит:

- собственный локальный ID tracklet;
- `person_id`, если identity уже присвоена;
- Kalman state;
- текущий и сглаженный bbox;
- путь движения;
- feature/color/face/shape банки;
- счетчики hits, missed, feature updates.


11. Kalman Filter

Состояние Kalman Filter имеет 8 измерений:

- `cx`, `cy`
- `w`, `h`
- `vx`, `vy`
- `vw`, `vh`

То есть модель хранит не только центр и размер рамки, но и их скорости.

Что это дает:

- между detect-кадрами можно прогнозировать движение;
- краткие пропуски детекций не приводят к мгновенной потере tracklet;
- визуальное движение bbox становится плавнее.

Обновление работает в два режима:

- `predict`
  используется на каждом кадре, чтобы продолжить движение трека вперед;

- `correct`
  вызывается на detect-кадрах, когда есть новое измерение из детектора.


12. Сглаживание bbox

После предсказания или измерения bbox не принимается "как есть".
Дополнительно используется сглаживание:

- отдельно для центра;
- отдельно для размеров;
- с разными коэффициентами для prediction-step и measurement-step.

Параметры:

- `prediction_center_alpha`
- `prediction_size_alpha`
- `measurement_center_alpha`
- `measurement_size_alpha`
- `size_jump_ratio`
- `center_deadzone_px`
- `size_deadzone_px`

Практический смысл:

- убрать дрожание рамок;
- не реагировать слишком резко на шум детектора;
- ограничивать нереалистичные скачки размеров.


13. Сопоставление detection <-> tracklet

Сопоставление выполняется не жадно, а глобально через Hungarian assignment.

Стоимость пары detection-tracklet строится из нескольких компонентов:

- IoU между predicted bbox и detection bbox;
- расстояние между центрами;
- similarity размеров;
- ReID appearance similarity;
- color similarity;
- при наличии face feature - face similarity.

Логика специально многокритериальная:

- motion alone ненадежен при пересечениях людей;
- appearance alone ненадежен при похожей одежде;
- color alone слишком слабый;
- комбинация дает более устойчивый cost.

Также в коде есть hard-gating:

- слишком далеко и слишком непохоже - пара запрещается;
- подтвержденный трек с плохой геометрией и слабым appearance не должен слипаться с чужой детекцией;
- если face similarity достаточно высокая, лицо получает повышенный вес.


14. Подтверждение и смерть tracklet

Каждый tracklet проходит стадии:

- birth:
  создается из новой детекции;

- tentative:
  еще не считается достаточно надежным;

- confirmed:
  после накопления `min_confirmed_hits`;

- lost/finished:
  после превышения `max_missed_detections`.

Дополнительный критерий readiness для передачи в identity layer:

- `identity_min_hits`
- `identity_min_feature_updates`
- `min_identity_box_area`

То есть identity не создается слишком рано.
Сначала tracklet должен:

- прожить несколько кадров;
- накопить достаточно appearance updates;
- быть достаточно крупным в кадре.


15. Удаление дублей среди tracklet

Даже после suppression детекций иногда могут родиться два tracklet для одного человека.
Поэтому tracker выполняет deduplication активных треков.

Используемые критерии:

- overlap рамок;
- nested overlap;
- similarity feature;
- similarity color;
- приоритет трека по hits, размеру, длине пути и числу пропусков.

После последних исправлений удаленный дубликат не исчезает бесследно:

- он возвращается как finished tracklet;
- `IdentityManager.finalize_tracklets()` успевает увеличить `tracklets_count`.

Это важно для корректной статистики в итоговом JSON.


16. Identity manager: долговременная память о человеке

`identity_gallery_tracking/identity_manager.py` хранит уже не временные tracklet, а persistent identities.

Каждая `IdentityRecord` содержит:

- текущий усредненный ReID feature;
- color gallery;
- face gallery;
- shape gallery;
- банки последних признаков;
- время первого и последнего наблюдения;
- число наблюдений;
- число завершенных tracklet, ассоциированных с identity;
- пройденную в пикселях дистанцию;
- trajectory points.

То есть identity - это не один вектор, а живая галерея признаков, обновляемая во времени.


17. Как identity создается и обновляется

Для каждого подтвержденного и готового tracklet система пытается:

1. сравнить его с уже существующими identity;
2. выбрать лучший кандидат;
3. проверить, достаточно ли он лучше второго кандидата;
4. если нет уверенного победителя - создать новую identity.

Пороговая логика нужна потому, что в re-identification слишком опасны ложные слияния.
Ошибочно объединить двух разных людей под одним identity ID хуже, чем временно выдать новый ID.

Ключевые параметры:

- `identity_reid_threshold`
- `identity_color_threshold`
- `identity_shape_threshold`
- `identity_face_threshold`
- `identity_match_margin`
- `identity_max_age`


18. Match score для identity

Система использует два режима оценки.

Режим 1: есть лицо

Если:

- у tracklet есть face embedding;
- у identity есть face gallery;
- face similarity достаточно высокая;

то итоговый score строится с доминирующим вкладом face similarity,
а appearance/color/shape играют вспомогательную роль.

Режим 2: лица нет

Тогда система полагается на:

- ReID appearance similarity;
- color similarity;
- shape similarity;
- возрастной штраф.

Возрастной штраф нужен для того, чтобы:

- старые давно не наблюдавшиеся identity не выигрывали слишком легко;
- недавние кандидаты были предпочтительнее при прочих равных.


19. Feature banks и momentum update

Identity не хранит один-единственный embedding.
Вместо этого используются:

- основной усредненный feature;
- банки recent features;
- банки recent color hist;
- банки recent face features;
- банки recent shape descriptors.

Обновление делается через экспоненциальное смешивание:

- `identity_feature_momentum`
- `identity_color_momentum`
- `identity_shape_momentum`

Плюсы этого подхода:

- identity медленнее реагирует на шум;
- но постепенно адаптируется к изменению позы, масштаба и освещения;
- банки позволяют искать не только по одному среднему вектору, но и по лучшим историческим примерам.


20. Почему identity и tracklet разделены по обновлению

После недавних исправлений проект обновляет identity только по реально наблюдаемым tracklet.

Это означает:

- Kalman prediction может двигать bbox на экране;
- но identity-логика не считает это полноценным новым наблюдением;
- `observations`, `last_seen_frame`, `distance_px` и route не растут без детекции.

Такой режим гораздо ближе к реальному смыслу данных в JSON.


21. Face backend

Модуль `identity_gallery_tracking/face_backend.py` опционален.

Он активируется только если доступны модели:

- YuNet для детекции лица;
- SFace для face recognition.

Последовательность работы:

1. из bbox человека вырезается crop;
2. внутри него ищется лицо;
3. среди найденных лиц выбирается лучшее по score * area;
4. лицо выравнивается через `alignCrop`;
5. вычисляется face embedding;
6. embedding нормализуется и используется в матчинге.

Почему face backend сделан optional:

- на многих сценах лицо часто не видно;
- не всегда есть нужные ONNX-модели;
- проект должен работать и без лицевой биометрии.

Но если face backend доступен, он существенно улучшает устойчивость identity
при похожей одежде и пересечениях людей.


22. Производительность и адаптивность

Проект содержит несколько встроенных оптимизаций.

1. Детекция не на каждом кадре

- уменьшает нагрузку на YOLO;
- повышает FPS.

2. Выбор кандидатов для дорогого ReID

Не каждая детекция отправляется в тяжелую сеть.
Алгоритм пытается выбрать только те боксы, где ReID действительно нужен:

- новый объект;
- объект без присвоенной identity;
- плохое геометрическое совпадение с существующим tracklet;
- достаточно крупный bbox;
- detect-pass, на котором пора обновить appearance.

3. Ограничение `max_reid_detections`

- спасает от ненужной перегрузки сцены при большом числе людей;
- позволяет удерживать более стабильную скорость.

4. Потоковый захват камеры

Для live-camera может использоваться отдельный поток чтения кадров.
Это уменьшает зависание камеры, когда основная обработка тяжелая.

Дополнительно в текущей версии реализован fallback:

- если в Python-среде не работает backend потоков;
- программа не падает;
- а переключается на синхронный захват камеры с предупреждением.


23. Оценка FPS

На экране показывается `FPS`, но его источник зависит от типа входа:

- для live-camera FPS оценивается по фактическому времени между чтениями;
- для видеофайла по возможности используется metadata fps;
- если metadata ненадежна, применяется оценка по timestamp/PTS.

Это дает более честную цифру, чем слепо доверять `CAP_PROP_FPS`.


24. Отрисовка

Модуль `identity_gallery_tracking/renderer.py` рисует:

- bbox;
- подпись `Track <id>` или `ID <person_id>`;
- статус `pending`, если identity еще не присвоена;
- траекторию движения;
- стрелку направления;
- панель с FPS, количеством tracklet и количеством идентифицированных людей;
- индикатор `REC`, если идет запись.


25. Что сохраняется по завершении

Каждый запуск создает отдельную папку в:

- `identity_tracking_output`

Внутри обычно находятся:

- `<source_name>_events.json`
  основной лог identities;

- `<source_name>_tracked.mp4`
  видео с визуализацией;

- `identity_001_route.png`, `identity_002_route.png`, ...
  маршруты для отдельных identities.

JSON содержит:

- `saved_at`
- `project`
- `session_duration_sec`
- `identities`

Для каждой identity записываются:

- `identity_id`
- `first_seen_frame`
- `first_seen_time_sec`
- `last_seen_frame`
- `last_seen_time_sec`
- `observations`
- `tracklets_count`
- `distance_px`
- `trajectory_points`
- `has_face_gallery`
- `has_reid_gallery`
- `route_image`


26. Построение route image

Для каждой identity создается отдельный PNG-маршрут:

- фон темный;
- первые сегменты темнее;
- более поздние сегменты светлее;
- старт помечается зеленой точкой;
- конец красной;
- последний сегмент может содержать стрелку направления.

Это полезно не только для красивой визуализации, но и для быстрой ручной диагностики:

- действительно ли траектория была непрерывной;
- не было ли резких телепортаций;
- не произошло ли ложного слияния двух людей в одну identity.

30. Ограничения и практические замечания

Несмотря на более правильную архитектуру, система не магическая.

Ограничения:

- без face backend она идентифицирует человека по внешности, а не по строгой биометрии;
- похожая одежда у двух людей может приводить к ошибкам;
- очень маленькие bbox дают плохой ReID;
- сильные перекрытия и резкие повороты корпуса ухудшают стабильность;
- при CPU-only режиме скорость может заметно падать;
- неверно подобранные thresholds могут либо плодить новые ID, либо сливать разных людей.

Практический вывод:

- tracking и re-identification здесь сделаны аккуратно и модульно;
- но качество результата все равно сильно зависит от сцены, камеры, освещения и подобранных параметров.


31. Краткий итог

`identity_gallery_tracking` - это не просто "рисование рамок".
Это составная система из нескольких уровней:

- person detection;
- Kalman-based local tracking;
- Hungarian global assignment;
- appearance re-identification;
- color and shape auxiliary matching;
- optional face verification;
- persistent identity gallery;
- визуализация и логирование маршрутов.

Главное достоинство проекта:

- он разделяет задачу "вести прямоугольник" и задачу "помнить человека".

Именно поэтому архитектура подходит не только для demo-трекинга,
но и как база для дальнейшего развития:

- более сильный ReID backbone;
- калибровка thresholds под конкретную сцену;
- более богатый debug mode;
- экспорт richer analytics;
- добавление UI или веб-панели поверх результатов.
